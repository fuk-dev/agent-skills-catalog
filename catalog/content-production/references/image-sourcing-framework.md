# 画像調達判断フレームワーク（2手段: メディアライブラリ / AI 生成）

> プロセス5で使用。各画像をどう調達するかを判断する。
>
> **本 fork の前提**: 実行環境 (Managed Agent) が扱える画像の調達手段は 2 つだけ。人手フロー (Dropbox / Google Drive 受け取り、ストックフォト購入、カメラマン撮影発注) は扱わない。人手前提の craft 知識は `GENERALIZATION_GUIDE.md` の削除必須パターンに従って本ファイルから除外している。

---

## 1. 2 つの調達手段

```
【手段1】既存素材 (実行環境のメディアライブラリ)
  └─ ユーザーが事前にアップロードした素材から選択
  
【手段2】AI 画像生成
  └─ 実行環境の AI 画像生成ツールで生成
```

以下は agent scope 外 (実行できないので判断・出力しない):

- ストックフォト購入 (iStock / Adobe Stock / Shutterstock / Unsplash / Pexels / 写真AC / Pixta 等の外部サービス直接参照)
- 新規撮影 (カメラマン手配・撮影発注)
- クライアントからの素材受け取り workflow (Dropbox / Google Drive 等) — この経路で入った素材は最終的に **メディアライブラリ経由** で提供される想定

外部 URL (例: `images.unsplash.com/...`) を **prompt 内で書かない**。仮に craft 知識として頭にあっても、実行環境に存在しない URL は幻覚として扱われる。

## 2. 判断フロー

画像の用途に応じて以下のフローで判断:

```
[画像の用途確認]
  ↓
[実物・実在が必要？]
  ↓ Yes → 施工事例・スタッフ・店舗・症例・物件・商品 等
  │       ↓
  │       [メディアライブラリに該当素材あり？]
  │         ↓ Yes → 【手段1: 既存素材】
  │         ↓ No  → ユーザーに「実物写真をメディアライブラリに
  │                 アップロードしてください」と依頼して停止
  │
  ↓ No → 抽象的イメージ・背景・アイコン・イラスト
        ↓
        [オリジナリティ or 特定コンセプト必要？]
          ↓ Yes → 【手段2: AI 生成】
          ↓ No  → 【手段1: メディアライブラリの汎用素材】
                    ↓ 該当なし → 【手段2: AI 生成】
```

**判断原則**: 実物・実在が必要な用途は AI 生成 NG (幻覚)。無ければユーザーに依頼して停止する。

## 4. 手段1: 既存素材（実行環境のメディアライブラリ）

### メリット

- 実物・実在の表現ができる
- 権利関係がクリア (アップロード時にユーザーが確認済み前提)
- 即座に使用可能

### デメリット

- 素材が無い場合は停止 (agent は撮影・購入・外部取得できない)
- 品質はアップロード元次第

### チェック項目

```
[ ] メディアライブラリに該当用途の素材があるか
[ ] 解像度は十分か（後述「6. 画像品質の基準」参照）
[ ] 用途と合っているか（業種・シーン）
[ ] 現状（店舗・商品）と一致しているか（古すぎないか）
```

### 使い方

1. 実行環境のメディアライブラリを参照
2. 用途に合う素材を選定
3. 該当素材が無ければ **ユーザーに「〇〇の画像をアップロードしてください」と依頼して停止**
4. 素材があれば適切な alt テキストと共にページに配置

**禁止**: メディアライブラリに無い場合に、agent の craft 知識で外部の URL (Unsplash / iStock 等) を書き出さない。実在しない URL の幻覚事故になる。

## 5. 手段2: AI 画像生成

実行環境が提供する AI 画像生成ツールを利用する。以下は craft 知識としての一般情報。

### 主要サービス（2026年時点の一般情報）

| サービス | 特徴 | 価格目安 |
|---|---|---|
| **Midjourney** | 高品質、芸術的 | 月額 $10-60 |
| **DALL-E 3** | ChatGPT 連携 | ChatGPT Plus 月額 $20 |
| **Stable Diffusion** | ローカル実行可、自由度高 | 無料〜月額 |
| **Adobe Firefly** | 商用利用安全、Adobe 連携 | 月額プラン |
| **Leonardo AI** | ゲーム系・イラスト系 | 無料〜月額 |

**注**: 上表は craft 参考。実際にどのサービスを叩くかは **実行環境の tool** で決まる。agent が「Midjourney で作ります」等と特定サービスを名指しで宣言しない。

### AI 生成が向いている用途

- **抽象的なイメージ画像**（ヒーロー画像等）
- **背景・パターン**
- **アイコン・イラスト**
- **架空のコンセプト表現**
- **OGP 画像のベース**

### AI 生成が不適切な場面（絶対NG）

- **実物・実在の表現**:
  - 施工事例
  - スタッフ写真
  - 店舗写真
  - 商品写真（実物）
  - 症例（医療）
  - 物件（不動産）
- **特定人物の表現**（肖像権侵害リスク）
- **ブランドロゴ・既存キャラクター**（著作権侵害リスク）

これらは実行環境のメディアライブラリから既存素材を使う (無ければ停止・ユーザー依頼)。

### 商用利用ガイド

#### 5-A. サービス別ライセンス詳細比較（2026年時点）

| サービス | 商用利用 | 著作権帰属 | 補償制度 | クレジット表記 | 特記事項 |
|---|---|---|---|---|---|
| **Midjourney** | 有料プランで可 | ユーザーに帰属 | なし | 不要 | 無料プランは商用不可。年商100万ドル以上の企業はProプラン以上必須 |
| **DALL-E 3** | 可 | ユーザーに所有権 | なし | 不要 | OpenAIの利用規約に準拠。API経由も同様 |
| **Stable Diffusion** | 可（モデル依存） | モデルライセンス次第 | なし | モデル依存 | SDXLはApache 2.0で商用可。サードパーティモデルは個別確認必須 |
| **Adobe Firefly** | 可（商用安全） | ユーザーに帰属 | **あり**（IP補償） | 不要 | Adobe Stock・パブリックドメインのみで学習。最も安全 |
| **Leonardo AI** | 可（プランによる） | ユーザーに帰属 | なし | 不要 | Free プランは制限あり。商用は Apprentice 以上推奨 |
| **Canva AI** | 可（Proプラン） | Canva規約に準拠 | なし | 不要 | Canva Pro内での生成は商用可。無料プランは制限あり |

#### 5-B. 商用利用判断フロー

```
[生成画像を使いたい]
  ↓
[商用サイト・広告等で使用？]
  ↓ Yes
  ↓
[有料プランで生成？]
  ↓ Yes → 各サービスの利用規約に従って使用可
  ↓ No → ⚠️ リスクあり（Adobe Firefly 等の安全なサービスを推奨）
  ↓
[人物・実在物を含む？]
  ↓ Yes → 肖像権・パブリシティ権注意（実在人物は避ける）
  ↓ No → 使用可
  ↓
[ブランドロゴ・キャラクターを含む？]
  ↓ Yes → 商標権侵害の可能性、使用不可
  ↓ No → 使用可
```

#### 5-C. 日本法でのAI生成画像の著作権整理

- **文化庁見解（2024年）**:
  - AI生成物単体では**著作権は発生しない**（人間の創作的寄与が必要）
  - プロンプトを工夫し、選別・加工した場合は**著作権が発生しうる**
  - 学習データに他人の著作物が含まれる場合、生成物が**類似すれば著作権侵害の可能性**
- **実務対応**:
  - AI生成であることを社内で記録
  - 既存作品と似すぎていないか確認
  - 有名キャラクター・実在人物を明示的に生成しない
  - 商用利用は Adobe Firefly のような**補償制度あり**サービスを優先

#### 5-D. AI生成画像の開示義務に関する動向

- **EU AI Act（2026年施行）**: 生成AI利用の明示が必要になる可能性
- **Google E-E-A-T**: AI生成コンテンツは減点対象になりうる、人間の監修・追加情報が必要
- **国内広告業界**: 一部で自主的な開示ガイドライン策定の動き

**推奨**: 現時点で法的義務はないが、以下の場合は開示を検討
- サイトの信頼性が特に重要な業種（医療・士業・金融）
- お客様の声・実績等、真実性が求められるコンテンツ（そもそもAI生成不適切）
- ブランド価値として透明性を重視する場合

#### 5-E. 実務チェックリスト（生成→確認→記録→使用の4ステップ）

**Step 1: 生成前チェック**
```
[ ] 使用目的が商用・非商用のどちらか明確
[ ] 使用するサービスの現在のプラン（無料/有料）
[ ] 現在のプランで商用利用可能か
[ ] 実在人物・特定ブランドを含まないプロンプト設計
```

**Step 2: 生成後チェック**
```
[ ] 既存の有名作品と類似していないか（画像検索で確認）
[ ] 人物の顔が特定できる有名人と似ていないか
[ ] ブランドロゴ・キャラクターが誤って含まれていないか
[ ] 6本指・不自然な体の一部がないか
[ ] テキスト（看板・書籍等）が意味不明になっていないか
[ ] 差別的・不快な表現がないか
[ ] 商用利用可のプランで生成したか
```

**Step 3: 記録・保存**
```
[ ] 使用したサービス・プランを記録
[ ] プロンプトを記録
[ ] 生成日時を記録
```

**Step 4: 使用**
```
[ ] alt テキスト適切
[ ] 誤解を招く配置になっていないか（実物写真かのように見せていないか）
```

### AI 画像の品質チェック

```
[ ] 解像度は十分か（Web用: 1920px 幅以上）
[ ] 歪みはないか（手指・顔の破綻）
[ ] 不自然な要素はないか（意味不明な文字、実在ロゴの混入）
[ ] ブランドトーンと合うか
[ ] 業種にふさわしいか
```

## 6. 画像品質の基準（両手段共通）

### 解像度

- **最低**: 1920x1080 px（FullHD）
- **推奨**: 2560x1440 px 以上

### ファイル形式

| 用途 | 推奨形式 | 代替 |
|---|---|---|
| Web 掲載（一般） | WebP | JPEG |
| ロゴ・アイコン | SVG | PNG（透過） |
| 写真（一般） | WebP | JPEG |
| イラスト | SVG / WebP | PNG |
| アニメーション | WebP（アニメ） | GIF / MP4 |

### 圧縮

- **JPEG**: 品質 80-90%
- **WebP**: 品質 85%
- **PNG**: 透過必要時のみ（JPEG より重い）

### ファイル名

統一した命名規則:

```
[section]-[purpose]-[number].[ext]

例:
- top-hero-01.webp
- service-gaiheki-01.webp
- works-case001-before-01.webp
- works-case001-after-01.webp
- staff-ceo-01.webp
- og-image-default.jpg
```

## 7. 画像のブランド統一

サイト全体で画像のトーンを揃えることが重要:

### 統一すべき要素

- **明るさ**: 全体的に明るめ / 暗め
- **彩度**: 高め / 低め
- **コントラスト**: 強め / 弱め
- **色調**: 暖色系 / 寒色系
- **構図**: 中心配置 / 三分割法

### 統一方法

1. **既存素材の事後編集**: 色調補正で統一
2. **AI 生成は同じプロンプトスタイル**: スタイル指定を固定して連続生成

## 8. 画像の権利管理

### 確認必須項目

| 項目 | 確認内容 |
|---|---|
| 著作権 | 誰が持っているか、使用許諾はあるか |
| 肖像権 | 人物が写っている場合の本人許諾 |
| プロパティライセンス | 特定建物・商品等 |
| 商標権 | ロゴ・商品名の使用 |

### よくあるトラブル

1. **スタッフ退職後の写真継続使用**: 退職時に使用可否を契約明記 (ユーザー側の運用課題)
2. **施工事例の施主許諾**: 施工契約時に「事例として公開可」の承諾を得る (ユーザー側の運用課題)
3. **AI 生成画像の著作権**: サービスによって異なる、最新情報確認 (§5-A / §5-C を参照)
