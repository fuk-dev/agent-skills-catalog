# AI検索時代のコンテンツ原則（全業種共通）

> 原稿の企画・執筆・レビューのすべてで参照する横断リファレンス。
> 「何を書くか」を決める段階（企画）、「どう書くか」（執筆）、「これでいいか」（レビュー）の3局面で使う判断軸を提供する。

---

## 0. 大前提（これを外すと全部ズレる）

AI検索（Google AI Overview / ChatGPT / Claude / Perplexity 等）の時代でも、本質は変わらない。

**「AIに評価されるためのサイト」ではなく「ユーザーに価値を提供するサイト」を作る。**

AIの評価はその結果としてついてくるものであって、目的ではない。原稿を企画・執筆・レビューするときは常に「これは読み手の意思決定を助けるか」を先に問う。AI最適化の小手先（不自然なチャンキング、キーワード詰め込み、Q&A量産）は、この問いに「No」なら全部無駄になる。

なぜこれが効くのか：検索エンジンもAIも、今後は**その会社・店舗・組織にしか存在しない情報**を持つサイトを優先的に評価する方向に進んでいる。誰でも作れる情報をいくら積んでも差はつかない。差がつくのは一次情報の量と質だけ。

---

## 1. コモディティコンテンツ vs 独自コンテンツ

### コモディティコンテンツとは

他社でもAIでも作れる一般情報。

例：「○○とは」「○○のメリット」「○○の種類」「○○の流れ」「○○の料金相場」「○○の基礎知識」

### 方針：削除はしない。主役にしない。

一般情報には2つの役割がある — 検索意図のカバー（「○○とは」で来る人がいる）と、専門用語の前提共有。だから消す必要はない。最低限は整備する。

ただし**サイトの主役にしてはいけない**。力を入れるべきは独自情報。原稿の企画段階で、コモディティ記事ばかりの計画になっていたら設計をやり直す。

### 判断の目安

| 種別 | サイト内の役割 | 投下する工数 |
|---|---|---|
| コモディティ | 検索意図カバー・前提共有 | 最低限（テンプレ流用可） |
| 独自情報 | サイトの主役・差別化の核 | 最大限（取材・データ収集に投資） |

---

## 2. 一次情報を増やす（最重要）

AIが最も苦手なのは、現場でしか手に入らない情報。ここが勝負どころ。

- **現場情報** — 実際の作業・対応の様子
- **実体験** — やってみて分かったこと
- **独自データ** — 自社でしか持っていない数字
- **顧客の声** — 第三者の生の評価

### 積極的に掲載すべき4カテゴリ

**① 実績**
導入事例 / 施工事例 / 症例紹介 / 成功事例 / 事例インタビュー

**② 顧客の声**
レビュー / アンケート / 満足度調査 / 利用後の感想

**③ 現場の知見**
実際によくある相談 / 現場で起きるトラブル / 対応事例 / 実務ノウハウ

**④ 独自調査**
自社調査データ / 独自集計 / 市場分析 / 顧客分析

### 企画時の問い

新しい原稿を計画するとき、「この内容はAIが3秒で生成できるか？」を問う。できるなら一次情報が足りていない。「うちの現場でしか書けない要素」を1つ以上必ず入れる。

---

## 3. 経験をコンテンツ化する

良いコンテンツとは、**専門家や現場担当者しか書けない内容**。業種別の具体例：

| 業種 | 経験をコンテンツ化した例 |
|---|---|
| 飲食店 | 人気メニューランキング、実際のお客様の反応 |
| 不動産 | 成約事例、地域ごとの傾向 |
| 建設業 | 工事事例、施工時の注意点 |
| 医療 | よくある相談、症状別の来院傾向 |
| 士業 | 実際に多い相談事例、解決事例 |

執筆時のコツ：抽象論（「丁寧に対応します」）ではなく、**実際に起きた具体的な場面**を書く。「先月、〇〇という相談があり、△△の方法で解決した」のように。これは `industry-presets.md` のNG表現排除と完全に同じ思想。

---

## 4. テクニカルSEO（土台）

順位を上げる施策ではなく、**検索エンジンが正しく理解できる状態を作るための基盤**。コンテンツがどれだけ良くても、土台が崩れていれば評価されない。

必須項目：robots.txt / sitemap.xml / canonical / 301リダイレクト / 404対応 / モバイル対応 / Core Web Vitals / SSL / パンくずリスト

→ 標準構成（Astro + Cloudflare Workers）では `astro-base-theme` が大半をカバー。原稿側ではこれらの存在を前提に、内部リンク・パンくず階層と整合する見出し構造で書く。

---

## 5. 構造化データ

検索エンジン・AIに「このページは何について書いているのか」を正確に伝えるため。

実装推奨：Organization / LocalBusiness / Article / FAQPage / BreadcrumbList / Review / Service / Product / Person

→ 実装は `astro-base-theme` の業種別プリセット。原稿側では、構造化データに流し込める形（事例なら成果の数字、レビューなら評価者・日付、FAQなら質問と回答の対）でデータを揃えて納品する。

---

## 6. 評価指標：順位だけを追わない

| 悪い指標（手段の指標） | 良い指標（成果の指標） |
|---|---|
| PV / 表示回数 / 検索順位 | 問い合わせ数 / 来店数 / 来院数 / 資料請求数 / 見積依頼数 / 商談数 / 売上 |

順位やPVは手段であって目的ではない。原稿のCV導線（`page-copy-templates.md` の鉄則）はこの成果指標に直結させる。SEO記事の効果測定（`seo-annual-balance-sheet.md`）でも、流入数だけでなくCVへの寄与を見る。

---

## 7. AI向けの不自然な最適化をしない

以下はやらない。ユーザー価値を損なうだけで、AIにも見抜かれる。

- **不要なチャンキング** — 文章を不自然に細かく区切る
- **無意味なQ&A量産** — 「Q: ○○とは？ A: ○○とは〜」を機械的に大量生成する（※読み手の実際の疑問に答えるFAQは別。これは「価値のない量産」を指す）
- **AI向けキーワード詰め込み** — 不自然な繰り返し
- **過剰なHTML最適化** — 文法100点満点を目的化する
- **AI専用ページ作成** — ユーザー価値のないページの量産

判断基準：「これは読み手のためか、AIに媚びるためか」。後者なら削る。

## 8. AI生成コンテンツの扱い

### 推奨：AIは補助ツール

構成作成 / 下書き作成 / 要約 / 誤字脱字チェック — ここまではAIに任せてよい。

### 禁止

- AIが生成した内容を未確認で公開する
- 事実確認をしない
- 大量ページを生成する
- コピペで運用する

### 原則

**AIは執筆補助。責任者は人間。** 一次情報・固有名詞・数字・法令に関わる記述は必ず人間が確認する。特に医療・士業・不動産など法令配慮業種では、AI下書きをそのまま出さない（`industry-presets.md` の法令配慮を必ず通す）。

## 9. コンテンツ優先順位（S/A/Bランク）

原稿の企画・工数配分はこのランクに従う。

### Sランク（最優先・独自情報）
実績 / 事例 / 顧客の声 / 独自調査 / 独自データ / 専門家見解 / 現場レポート

### Aランク（中優先・準独自）
比較記事 / ノウハウ記事 / FAQ / 業界解説

### Bランク（最低限・コモディティ）
一般知識記事 / 用語解説

**工数配分の目安**：S に最大、B は最低限。Bばかりの計画はAI検索時代に負ける。

## 10. 全業種共通のコンテンツ設計（7要素）

サイト全体・主要ページは、この7要素を意識して構成する。

| # | 要素 | 役割 | 対応する独自情報 |
|---|---|---|---|
| ① | 基礎情報 | サービス説明 | — |
| ② | よくある質問 | 顧客の疑問解消 | 現場の知見（§2） |
| ③ | 実績・事例 | 信頼獲得 | 実績（§2 Sランク） |
| ④ | 顧客の声 | 第三者評価 | 顧客の声（§2 Sランク） |
| ⑤ | 独自データ | 専門性証明 | 独自調査（§2 Sランク） |
| ⑥ | 専門家の見解 | 権威性強化 | 専門家見解（§9 Sランク） |
| ⑦ | 問い合わせ導線 | CV獲得 | §6 の成果指標に直結 |

③〜⑥がサイトの差別化の核。①②は土台。⑦で回収する。

---

## まとめ：最終結論

最も重要なのは **「誰でも作れる情報を増やすこと」ではなく、「その会社・店舗・組織にしか存在しない情報を蓄積し発信すること」**。

全業種共通で、**一次情報・実績・顧客の声・現場経験をいかにコンテンツ化するか**が最重要課題。原稿を企画・執筆・レビューするときは、毎回この一点に立ち返る。
